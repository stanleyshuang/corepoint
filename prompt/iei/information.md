## 弱點修復模式
由於部分弱點源自上游供應商（例如 CPU 或晶片組廠商），IEI 在弱點處理流程中，可能依據上游供應商提供的修補或緩解指引，透過更新自家 BIOS、firmware、driver 或 software application，向客戶提供 mitigation 方案。
例如，當特定 CPU 的 CVE 公布後，IEI 可能透過整合上游提供的 microcode 或相關設定調整至 BIOS 更新中，提供客戶緩解措施。

### 供應商來源弱點之處理說明

當產品弱點來源涉及上游供應商所提供之元件、模組或技術（如 CPU、晶片組、BIOS 元件、driver、第三方 software library 等）時，本公司應依弱點性質、產品影響範圍及供應商所提供之技術指引，評估並採行適當之因應措施。

若上游供應商已提供修補程式、microcode、firmware、driver 更新、設定調整建議或其他緩解措施，本公司得透過更新自有產品之 BIOS、firmware、driver、作業系統設定或 software application 等方式，整合相關修補或緩解措施，並提供客戶執行，以降低弱點所造成之風險。

例如，當特定 CPU 或晶片平台之弱點已被公開並取得 CVE 編號時，本公司得依供應商公告之修補或緩解指引，透過 BIOS 更新整合相關 microcode、調整平台設定，或提供其他對應之軟體／韌體更新，以作為客戶端之 mitigation 方案。

前述措施之目的，在於於正式修補完成前或無法立即更換受影響元件之情況下，降低弱點被利用之可能性、縮小攻擊面，或減輕其對產品運作與資訊安全之影響。

### 條文式版本

**X.X 供應商來源弱點之處理**

1. 當弱點涉及上游供應商提供之硬體、韌體、driver、函式庫或其他第三方元件時，應確認其對本公司產品之影響範圍。
2. 本公司應依供應商提供之安全公告、修補程式、技術文件或緩解指引，評估可行之處理方式。
3. 必要時，本公司得透過更新 BIOS、firmware、driver、software application 或調整系統設定等方式，整合並提供客戶可執行之 mitigation 方案。
4. 前述 mitigation 方案得用於降低弱點風險、限制攻擊條件、縮小影響範圍或減輕潛在損害。
5. 若供應商尚未提供正式修補方案，本公司應持續追蹤供應商處理進度，並視情況提供暫行性風險降低措施。

摘要：
你這段內容在 PSIRT 程序書中可以表述為「**對於供應商來源弱點，公司依上游指引評估影響，並透過 BIOS、firmware、driver 或軟體更新提供 mitigation**」；重點是寫成「整合上游修補或緩解措施」，不要寫成像是公司自行修復 CPU 弱點本體。

# 三、直接可用的公告句型

下面這些句子可以直接抽換使用。

## 1. 弱點來源描述

> 本弱點源自上游供應商提供之硬體或軟體元件，IEI 已評估其對相關產品之影響，並依供應商提供之技術指引實施對應處置。

## 2. IEI 緩解方式描述

> IEI 已透過更新 BIOS、UEFI firmware、ME firmware、driver 或相關 software package，整合上游供應商提供之修補或緩解措施，以降低弱點被利用之風險。

## 3. CPU / BIOS 情境描述

> 對於涉及特定 CPU 平台之弱點，IEI 可能透過 BIOS 更新整合供應商提供之 microcode 或相關設定調整，作為客戶端之 mitigation 方案。

## 4. 非完全修補時的保守寫法

> 本更新旨在降低該弱點之利用風險與潛在影響，使用者仍應依建議完成所有相關更新與安全設定調整。

## 5. 無法立即更新時

> 如客戶暫時無法立即完成更新，建議先採取 IEI 提供之暫行性風險降低措施，以減少暴露面與攻擊可能性。

# 五、程序書／公告常用詞建議

建議你在正式文件中多用下面這些詞，比較專業且安全：

* **源自上游供應商元件**
* **依供應商技術指引**
* **整合修補或緩解措施**
* **提供 mitigation 方案**
* **降低風險 / 縮小攻擊面 / 減輕潛在影響**
* **暫行性風險降低措施**
* **於可控產品層面交付更新**

比較不建議直接寫：

* 「IEI 修復 CPU 弱點」
* 「IEI 解決供應商弱點本體」

因為這會讓人誤解成根本修補是 IEI 自行完成。
