# L3-01：弱點通報受理、ACK 與對外溝通（含 Business Partner 協調）

- **文件屬性**：作業程序（Level 3）
- **適用標準**：ISO/IEC 29147（接收通報、與通報者溝通、協調揭露）、ISO/IEC 30111（Receipt / Verification / Confidentiality、避免過早揭露）
- **版本**：v0.2（草案）
- **文件擁有者**：PSIRT
- **相關文件**：L1（政策 v0.2.1）、L2（流程）、L3-02（分流）、L3-03（分析/嚴重性）、L3-04（Jira 管理）、L3-09（揭露/通報）

## 1. 目的（Purpose）

定義弱點通報受理到結案期間，PSIRT 對外溝通（通報者 / 客戶 / 合作夥伴 / 協調者）的標準做法，確保：

- 通報可被快速受理、追蹤與回覆，並符合對外承諾時程。
- 全程資訊受控（need-to-know），避免過早揭露與可武器化細節外流。
- 對外訊息一致、可稽核、可回溯（與內部分析 / 修補 / 揭露一致）。
- Business Partner（合作夥伴 / 外部協力開發商）溝通具單一窗口與時程協調機制。

## 2. 適用範圍（Scope）

適用於所有弱點來源：

- 外部研究員 / 客戶 / 第三方通報
- 合作夥伴 / 供應鏈通知（Business Partner）
- 內部發現（若需對外溝通或跨組織協調）

## 3. 名詞（Terms）

### ACK（Acknowledge）
對通報者確認已收到弱點報告的回覆。

### Update
案件處理期間，對通報者提供的進度更新（不揭露可武器化細節）。

### CVD
Coordinated Vulnerability Disclosure（協調式揭露）。

### Embargo
受控期間；在修補 / 緩解完成前限制公開細節之協議 / 安排。

### Business Partner
合作夥伴 / 外部協力開發商 / 供應鏈相關對象。

### 受控共享（Controlled Sharing）
need-to-know 原則下的最小必要資訊共享。

## 4. 對外溝通總原則（Governing Principles）

1. **單一對外窗口**  
   技術與安全面向之對外溝通由 **PSIRT** 統一窗口；涉及交付 / 上架時程由 **DPM** 協調，但對外口徑仍需 PSIRT 對齊。

2. **最小必要揭露**  
   在風險可控前，不提供可直接用於利用的細節（完整 PoC、可複製攻擊鏈、精準 exploitable payload 等）。

3. **一致性與可回溯**  
   對外內容需可回溯至內部分析 / 驗證與決策摘要（寫回 Jira Task）。

4. **尊重通報者與信用**  
   在不影響安全 / 法遵 / 保密下，提供合理更新與致謝安排（致謝規則由 L3-09 控制）。

5. **時效承諾**  
   遵循 L1 對外承諾：**收到報告後 3 個工作天內完成 ACK**。

## 5. 受理與追蹤（Intake & Tracking）

### 5.1 正式受理渠道（不列工具細節）

- 對外入口：`security@qnap.com`（主要通報信箱）、官方安全通報表單（如網站表單）、`/.well-known/security.txt` 所載資訊；必要時提供 PGP 公鑰。
- 內部 / 供應鏈：客服安全通報、合作夥伴通報渠道、CERT / 協調者對接。

接收弱點通報後，PSIRT 應立即建立追蹤主檔（Jira Task；細節見 L3-04），並將通報原文 / 附件以受控方式保存、標記敏感度。

### 5.2 監控與避免漏接（值班 / 備援）

- `security@qnap.com` 與表單須設定監控與超時提醒；責任名單由 PSIRT 維護。
- 若主要窗口無法回應，備援窗口需在 SLA 內代為 ACK；所有交接須留痕。
- 若使用自動 ACK（如 Salesforce），需每季檢查設定並留存設定截圖 / 測試結果（客觀證據）。

### 5.3 最小建案資訊（Minimum Intake Data）

PSIRT 需收斂以下最小資訊（不足亦可先 ACK，但需提出補件）：

- 通報者聯絡資訊（或匿名識別）
- 受影響產品 / 服務與版本（可 Unknown）
- 弱點描述與預期影響
- 重現方式（PoC / 步驟）或替代證據
- 是否已公開 / 是否計畫公開（對揭露時程影響重大）
- 是否觀測到在野利用（如有證據更佳）

### 5.4 外部回報欄位模板（最小集合，可用於表單 / 手動回報）

- 通報者聯絡方式（email / PGP，如適用）
- 受影響產品 / 版本 / 環境
- 問題描述與影響（CIA）
- 重現步驟或證據（PoC、Log、截圖）
- 是否已公開或預計公開時程
- 是否有在野利用跡象

## 6. ACK（確認已收到）規範

### 6.1 時效（SLA）

- **必須在 3 個工作天內完成 ACK**（L1 承諾）。
- 若遇到例外（如法遵限制、需先判定敏感度 / 受控共享範圍），仍需在期限內提供最小 ACK，並給下一次更新節點。

### 6.2 ACK 內容最小集合（Minimum Content）

ACK 應至少包含：

1. 已收到確認（含參考識別碼或案件代碼）
2. PSIRT 聯絡窗口
3. 後續處理節點（下一次更新時間點或區間）
4. 保密 / 受控揭露提醒（避免公開可武器化細節）
5. （如需）補件清單（最小化且具體）

### 6.3 ACK 範本（Template）

以下為可直接使用的範本（依個案裁切）。不包含敏感技術細節。

#### ACK（一般版本）

您好，感謝您的弱點通報。我們已確認收到並已啟動內部追蹤處理（參考識別碼：`{REF}`）。我們將在完成初步評估後提供下一次進度更新。為確保使用者安全，建議在修補 / 緩解完成前，暫勿公開可直接利用的技術細節；若您有預計公開時程，也請告知以利我方協調。若方便，請補充以下資訊以協助我們快速驗證：`{REQUESTED_INFO_LIST}`  
謝謝您  
PSIRT

#### ACK（敏感 / 疑似事件版本：受控）

您好，感謝您的通報。我們已確認收到並已啟動內部處理（參考識別碼：`{REF}`）。由於此議題可能涉及較高風險情境，我們將以受控方式進行驗證與處置，並於 `{NEXT_UPDATE_WINDOW}` 提供下一次進度更新。為降低風險，懇請在我們完成風險控管與必要修補前，暫勿公開可直接利用的細節（PoC、payload、完整攻擊鏈等）。  
謝謝您  
PSIRT

## 7. 更新時程（Update Cadence）與回覆內容控管

### 7.1 Update 最小集合（Minimum Content）

更新應包含：

- 目前階段（Triage / Analysis / Remediation / Disclosure）
- 已完成事項（不含可武器化細節）
- 下一步與預計節點（日期 / 區間）
- （如適用）需通報者補充資訊項目
- （如適用）對外揭露協調需求（embargo、同步發布）

### 7.2 回覆禁區（Do-Not-Disclose）

在未修補 / 未控風險前，避免提供：

- 可直接利用的 payload、完整 exploit code
- 精準可複製的攻擊鏈與關鍵參數
- 未公開修補的精準差異點（可能導致 N-day weaponization）

### 7.3 Update 頻率（量化基準）

- **V5 / V4**：每個工作天更新一次（即使無重大進度也需回覆「狀態未變 / 下一節點」）。
- **V3 / V2 / V1**：依進度節點更新（至少於分流完成、分析結論、修補 / 緩解決策、發布 / 結案時回覆）。

## 8. 對外溝通分類（Audience-specific Guidance）

### 8.1 與通報者（研究員 / 客戶）溝通

優先目標：確認有效性、取得必要資訊、協調揭露時程與致謝方式。

若通報者要求公開：  
PSIRT 應回覆政策原則（CVD、使用者安全、受控揭露），並提出合理更新節點與同步發布方案（由 L3-09 控制）。

### 8.2 與客戶溝通（非揭露階段）

- 若尚未公告但需提供緩解：僅提供必要緩解 / 設定指引，不提供可武器化細節。
- 若涉及事件並行：由 PSIRT 與 IR 對齊口徑後再對外回覆。

### 8.3 與協調者（Coordinator）溝通（如適用）

適用於多方受影響、需跨供應鏈協調之議題。  
PSIRT 可在 need-to-know 原則下提供必要資訊，並協調同步發布 / embargo。

### 8.4 與 Business Partner（合作夥伴 / 外部協力開發商）溝通

流程層原則：技術與敏感資訊由 PSIRT 控管；交付時程由 DPM 主責窗口（既定決策）。

#### 必遵循原則

1. **單一技術窗口**：PSIRT 作為技術資訊共享與敏感度控管窗口。
2. **交付 / 上架窗口**：DPM 作為交付時程與上架 / 發布協調窗口。
3. **受控共享**：只分享完成修補 / 交付所需最小資訊，並要求合作夥伴同等保密。
4. **同步時程**：若需同步發布 / embargo，必須在主追蹤主檔留痕並明確時點。

#### Business Partner 溝通範本（摘要版）

您好，我們正在就一項安全議題進行受控協調。為協助您完成修補 / 交付，我們提供以下最小必要資訊：`{MIN_REQUIRED_INFO}`  
請注意：此資訊屬敏感內容，僅限需要知道的人員使用，請勿對外公開或擴散。若您有預計交付時程或同步發布需求，請回覆：`{DELIVERY_SCHEDULE_REQUEST}`  
謝謝  
PSIRT（技術協調窗口）  
DPM（交付時程窗口）

## 9. 對外承諾與例外處理（Commitments & Exceptions）

### 9.1 對外承諾（最低）

- 3 個工作天內 ACK（必須達成，除非有例外核准留痕）。
- 更新時程依 §7 建議基準執行，若調整需留痕理由。

### 9.2 例外情境（需留痕）

- 法遵 / 調查限制（例如需配合執法或合約限制）
- 事件並行導致資訊高度敏感（需延後提供細節）
- 通報資訊不足且通報者無法補件：PSIRT 發出補件請求後 **14 個工作天內無回覆**，可依 L3-04 ABORT 規則中止（需記錄請求與結果）
- 合作夥伴交付不可控導致時程變更（由 DPM 留痕交付原因，PSIRT 留痕對外口徑調整）

## 10. 留痕與可稽核性（Record Keeping）

### 10.1 必要留痕位置

- 所有對外溝通（ACK / 更新 / 協調重點）需以「摘要」回填至 Jira Task（避免散落）。
- 重要溝通附件（必要時）可附於主追蹤主檔，並標記敏感度。

### 10.2 必填留痕要點（Minimum Records）

- ACK 發送日期、內容摘要、參考識別碼
- 每次更新日期、更新摘要、下一節點
- 任何揭露 / 同步發布承諾或限制（embargo）
- 例外處理原因與核准資訊

對應關係：L3-04 vNext 已覆蓋工單欄位與最小證據要求；本文件僅定義溝通留痕的內容項目。

## 11. 稽核檢核點（Audit Checkpoints）

1. 是否在 3 個工作天內完成 ACK（有日期與摘要）。
2. 更新頻率是否符合 §7.3（V5 / V4 每工作天；V3 以下依節點更新），或具例外核准留痕。
3. 是否存在合理的更新時程或例外核准留痕。
4. 是否避免過早揭露可武器化細節（抽查對外內容）。
5. Business Partner 案件是否有受控共享與交付時程紀錄（PSIRT + DPM 角色清楚）。
6. 溝通摘要是否可回溯到分流 / 分析 / 修補 / 揭露結果（與 L3-02 / L3-03 / L3-04 / L3-09 一致）。
